API (Aplication Programming Interface)
Uygulama Programlama Arayüzü

Backend denildiginde akla Database (SQL,JDBC) ve API gelmelidir.

Bir uygulamanin fonksiyonlarina erisilip kullanilmasin saglayan bir arayüzdür.

Bir uygulamanin hem IOS'ta, hem Windows'ta, hem de Android isletim sisteminde kullanilabilmesi API sayesinde olur.

API'in kullanici arayuzu yoktur. (User Interface{UI})

UI daki kullanici adi ve sifre istenmesi gibi API'de de islem yapabilmek icin token almamiz gereklidir.

http (Hiper Text Transfer Protocol)

Client ve Server arasindaki request(telep) ve responce(cevap)lar ile gerceklesen bir protokoldür.
Bir url yazildiginda karsimiza istenilen sayfanin gelmesi.

API TESTING isleminde Dogrulama

  • Eldeki verilerle API'ye bir istek (requst) gönderilir.
  • API'den aldigimiz cevaplar(responce):
    1. http status code
    2. data

http status code

  • 1xx :Informational codes(Bilgi)
  • 2xx :Succes codes(basari)
  • 3xx :Redirect codes(yönlendirme)
  • 4xx :Client error codes(tarayici hatasi)
  • 5xx :Server error codes(sunucu hatasi)
Örnek olarak;
  • 200 (OK) Basarili bir sekilde size responce döndürüldü.
  • 201(CREATED) Olusturma islemi basarili bir sekilde gerceklesti.
  • 204 (No CONTENT) Isleminiz basarili sekilde gerceklesti ama icerik olmadigi icin bir sey döndürülemedi.
  • 400 (BAD REQUEST) Hatali bir istek gönderdiniz.
  • 403 (FORBINDEN) Erisim izniniz yok. (size yasak ancak sifreyle girilebilir.)
  • 404 (NOT FOUND) Istek bulunamadi
  • 500 (INTERNAL SERVER ERROR) Server'in o an ulasilmaz oldugunu ifade eder.

API Protokolleri

  1. SOAP
  2. REST

SOAP (Simple Object Access Protocol- basit nesne erisim protokolü)

  • Bir protokoldür.
  • XML tabanlidir. XML yardimiyla methodlar cagrilir. Gönderilen verilere cevap XML formatinda döndürür.
  • SOAP tabanli bir Web Service'in gönderilen XML verisini nasil yorumlayacaginin tanimlanmasi gerekir. Bu tanimlama WSDL (Web Service Description Language - Web servisleri tanimlama dili) ile yapilir.
  • Ön bellegi okuyamaz. (can not be cached)
  • Rest'e göre daha yavastir.
  • Rest'e göre daha güvenlidir.
  • Finans, iletisim, ödeme noktalarinda kullanilir.
  • Kullanici, datayi server'a iletmek isterse öncelikle SOAP standardlarina uygun hala getirmelidir.
  • getProductName seklinde kullanilir.

REST (Representational State Transfer)

Dört temel methodu vardir:
  1. GET : Talep edilen datalari getirir ve okuruz.
  2. POST : Gönderilen verileri olusturur.
  3. PUT : Gönderilen verileri günceller.
  4. DELETE : Talep edilen datayi siler.
  • Bu methodlara CRUD (Create, Read, Update, Delete) da denir.
  • Http methodlariyla calisan bir mimaridir.(Rest mimarisi)
  • Http methodlariyla tasarlandigi icin kullanimi daha kolaydir.
  • Rest mimarisini kullanan servislere Restful denir.
  • Ön bellegi okur. (can be cached)
  • Soap'a göre dah hizlidir.
  • Veriler XML, JSON (Java Script Object Notation), TEXT, HTML formatlarinda olabilir.
  • Rest mimarisinde islemler resource kavrami ile yapilir.
  • Resource ise URI (Uniform Resource Identifiers) ile tanimlanir.
  • URI; örnegin amazon sitesine gittik ve orada bir ürüne tikladik. Adres cubugundaki linkin tamamina URI diyoruz. (https://www.techproeducation.com/courses)
  • Methodlar icin istek URI ile ve web service http protokolüyle yapilir.
  • Sosyal medya,web chat, mobil uygulamalarda kullanilir.
  • .../procucts/name/1 URI'i seklinde kullanilir.
  • CRUD islemlerini Postman programinda yapabilmek icin https://reqres.in/ sayfasindan yardim alinabilir.

Interview Sorusu

PUT ile PATCH'in farki nedir?

Ikisi de UPDATE yapar ama PUT dedigimizde baskasi tarafindan POST ile olusturulmus kaydin tamamini degistiririz. Girmedigimiz tüm bilgilerin value'sunu null yapar.
PATCH ile ise sadece ilgili verideki ilgili bilgi degistirilmis olur ve diger tüm bilgiler degistirilmeden ayni kalir. (Yama)
PUT'un POST'tan farki ise;
PUT yaparken URI nin sonunda ID vardir.
POST yaparken ise herhangi bir ID göndermiyoruz, data gönderiyoruz ve ID sitem tarafindan olusturuluyor.

REST Request'in Parcalari

  • Request-Line :Istegin hangi url'ye yapilacagi endpointtir. (http bilgisi)
  • Header :Yapilan istegin parametrelerdir. Datanin boyutu vs(yazilmasi zorunlu degil)
  • Body :Server'da bir degisiklik yapacaksak kullanilir. (Post, put patch gibi islemler)

Kaynaga nasil esisebilecegimizi göreteren URI'lardir.

Bir API olusturdugumuzda kullanicilarin ulasabilmesi icin bu endpoint'i ve kullanilabilecek http methodlarini kullanicilara bildirmeliyiz.

  • www.techproeducation.com/courses : URN

Path ve Query Parametreleri

https://regres.in/api/users?id=1

path param

Bir kaynagi tanimlamak ya da daha ayrintili nesneler üzerinde hareket etmek istersek kullaniriz.

Bir url'de ?'nden önce gelen kisimdir.

query param

Ögeleri siralamak veya filtrelemek istersek kullanniriz.

Bir url'de ? 'nin sag tarafinda görünen kuyruk kismidir.

Query parametreleri Key-Value seklinde kullanilir.

API Testing

IntelliJ ile API testleri yapabilmek icin POM.xml'e io-rest-assured ve junit dependency'lerinin yüklenmesi gerekir.

API testlerini yaparken Gherkin dilini kullaniriz.
  • Given : Istenen Endpoint sorgusundan önce yapilacak geereklilikleri ifade eder. (Json formatinda getir)
    given(). accept("application/json")...
  • When : Kullanicinin aksiyonunu belirtir. (API testindeki get,put,patch gibi komutlari icerir.)
    String url="http://dummy.restapiexample.com/api/v1/employees";
    ...when().get(url);
  • Then : Ciktilari ifade eder. Response islemlerinde kullanilir. (Assert islemleri genelde then ile yapilir. statusCode=200)
    response.then(). assertThat(). statusCode(200). contentType(ContentType.JSON).statusLine("HTTP/1.1 200 OK"); //aceept ("application/json") da yazilabilir.
  • And : Coklu islem yapilacaksa kullanilir.
Yazdigimiz kelimelerin siralamasi ve yeri önemlidir. (Cucumber'dan farklidir)

Vazifemiz: API servisine bir Request (istek) göndermek, gelen responce'u Assert (dogrulamak) etmektir.

Iki cesit Assertion vardir;
  • Assert : Selenium'da kullanilan fonksiyonlari kullanabiliriz.
  • Matcher Class : Gelen responce'lari actual result olarak atayarak assert edebiliriz.
Gelen responce'lar belli bir formatta gelmektedir (xml, json vs). Java'nin bu formatlari okuyabilmesi icin dgeisim gerekmektedir.

DE-Serialization : Json formatindaki bir datanin Java formatina cevrilmesine denir.

Serialization : Java formatindaki bir datanin json formatina cevrilmesi islemidir (bu islem su an icin bizi ilgilendirmiyor).

API Testinin Basamaklari

  1. URL(Endpoint) belirlenmelidir.
    • String url = "https://restful-booker.herokuapp.com/booking/3";
  2. Beklenen sonuc (Expected Result) olusturulmalidir.
  3. Bir istek (Request) gönderilmelidir.
    • Request gönderirken Gherkin dilinden faydalaniriz.
      Response response=given().accept("application/json").when().get(url);
    • given() i import ederken mutlaka RestAssured.given(io.restAssured) kütüphansinden manuel import etmeliyiz.
    • Eger response'taki body'i yazdirmak istersek;
      response.prettyPrint();
    • Eger response'taki header'lari yazdirmak istersek;
      System.out.println(response.getHeaders());
    • Eger baska bilgiler isteniyorsa;
      response.getStatusCode();
      response.getContentType();
      response.getStatusLine(); seklinde cagrilabilir.
  4. Geribildirim (Actual Result) alinmalidir.
  5. Dogrulama (Assertion) yapilmalidir. Cesitli yöntemlerle yapilabilir.
    • Cucumber'daki Assert dogrulamasini kullanabiliriz;
      Assert.assertEquals(200,response.getStatusCode());
      Assert.assertEquals("application/json; charset=utf-8",response.getContentType());
    • API dogrulamasini kullanabiliriz;
      response.then(). assertThat(). statusCode(200). contentType(ContentType.JSON). statusLine("HTTP/1.1 200 OK");
      • Eger response body'de belli bir kelime icerdigini veya icermedigini test etmek istersek .asString() methodunu kullanarak response'u String'e cevirmeliyiz.
        Assert.assertTrue(response.asString().contains("Not Found"));
        Assert.assertFalse(response.asString().contains("API"));
    • Matcher Class yardimiyla;
      response.then().
      assertThat().
      statusCode(200).
      contentType("application/json").
      body("firstname", Matchers.equalTo("Susan")).
      body("lastname",Matchers.equalTo("Jackson")).
      body("totalprice",Matchers.equalTo(811)).
      body("depositpaid",Matchers.equalTo(true)).
      body("bookingdates.checkin",Matchers.equalTo("2021-08-11")).
      body("bookingdates.checkout",Matchers.equalTo("2021-10-18"));
      • Her defasinda Matchers yazmamak icin import'u;
        import static org.hamcrest.Matchers.*;
        sekline ceviriyoruz. Artik Matchers'siz yazilabilir;
        body("firstname",equalTo("Susan"),
        "lastname",equalTo("Jackson"),
        "totalprice",equalTo(811),
        "bookingdates.checkin",equalTo("2021-08-11"),

JsonPath

Json formatinda gelen bir dataya ulasmak ve manipule etmek icin kullanilir.

$ isareti dokümandaki tüm node'lari verir.

Child bölümlere ulasmak icin (.) kullanilir. (store.book)

Eger index belirtilmezse hepsini getirir.

Belirli bir dataya ulasmak istiyorsak index kullanmaliyiz.
  • store.book[1]
    ikinci siradaki kitabi getirir.
  • store.book[0,3]
    Ilk ve dördüncü siradaki kitabi getirir.
  • store.book[-1]
    Son kitaba ulasmak icin kullanilir.
  • store.book[*].author
    Tüm kitaplarin yazarlarina ulasmak icin kullanilir.
Ayrinlili bilgi icin https://jsonpath.herokuapp.com/ sitesi incelenebilir.

Matchers Class

  • body("key",Matchers.equalTo("Value")) Key olarak verilen degiskenin degerininn value'ya esit olup olmadigini kontrol eder.
  • body("data.id",Matchers.hasSize("value")) Key olarak verilen degiskenin sayisinin value'ya esit olup olmadigini kontrol eder.
  • body("data.employee_name",Mathers.hasItem("value")) Key olarak verilen degiskenin aldigi degerlerin icinde value'nun olup olmadigini kontrol eder.
    Eger value birden fazla ise hasItems kullanip value'lari (,) ile yan yana yazabiliriz.

Örnek Soru

http://dummy.restapiexample.com/api/v1/employees url'ine accept type'i "application/json" olan GET request'i yolladigimda gelen response'un status kodunun 200 ve content type'inin "application/json" ve employees sayisinin 24 ve employee'lerden birinin "Ashton Cox" ve gelen yaslar icinde 21, 61, ve 23 degerlerinden birinin oldugunu test ediniz.

String url="http://dummy.restapiexample.com/api/v1/employees";
Response response=given().accept("application/json").when().get(url);
*response.prettyPrint();
response.then().assertThat().statusCode(200).contentType("application/json").
body("data.id", hasSize(24),
"data.employee_name",hasItem("Ashton Cox"),
"data.employee_age",hasItems(21,61,23));
